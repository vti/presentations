Test Driven development

Test::More?

Последовательность разработки

  1. Написание теста
  2. Тест не проходит
  3. Написание функционала
  4. Тест проходит
  5. Рефакторинг

Позитивные стороны

  1. Интерфейс "сам собой" продумывается
  2. Не реализуются ненужные требования
  3. Разработка системы маленькими шажками
  4. Вынуждает писать код модульным
  5. На ранних этапах выявляются ошибки проектирования
  6. Модуль работает
  7. Безопасный рефакторинг
  8. Mocks

Недостатки

  1. Высокий процент прохождения тестов может создать псевдо уверенность в коде
  2. Тесты нужно рефакторить также как и код
  3. Тесты могут быть хрупкими

Но недостатки ли это?

Хорошие практики

AAA
  • Arrange
  • Act
  • Assert

Пример

sub should_add_numbers {
    my $self = shift;

    # Arrange
    my $firstNumber  = 2;
    my $secondNumber = 3;
    my $calculator   = Calculator->new();

    # Act
    my $result = $calculator->calculate($firstNumber, $secondNumber);

    # Assert
    is($result, 5);
}
